Section · Coverage

Green ≠ tested code

Green tests only mean the tests you wrote pass. Coverage shows what your suite never touches.

1.1 · Green ≠ tested

A passing suite can miss whole branches

Here's a two-branch function whose only test hits one branch:

def apply_discount(price, is_member):
    if is_member:
        return round(price * 0.9, 2)   # branch A — the test hits this
    return price                       # branch B — never run

assert apply_discount(100, True) == 90.0   # ✓ all green
🔮
Predict: the test passes — is every line tested?

No — branch B never ran, and pass/fail can't show you that. You measure it.

1.2 · pytest-cov — the %, and the missing lines

Read the Missing column

--cov reports how much of each file ran; term-missing names the exact lines no test reached. --cov-branch also checks each if went both ways.

pytest --cov=catalog --cov-report=term-missing --cov-branch
Name                 Stmts   Miss Branch  Cover   Missing
---------------------------------------------------------
catalog/models.py       74      0     12   100%
catalog/server.py       38      8      0    79%   46-49, 54-57
catalog/storage.py      43     18      6    55%   39-47, 51-59
---------------------------------------------------------
TOTAL                  155     26     18    83%

83% isn't a failure — the Missing column found real gaps: the PATCH/DELETE routes and the CSV helpers.

1.3 · Coverage is a floor, not a target

100% ran ≠ 100% correct

100% means every line ran — not that it's correct (a line can run and still be wrong). Use coverage to find the untested branch that matters, not to chase a number.

def delete(self, product_id):
    if product_id not in self._items:            # is THIS branch tested? coverage tells you
        raise CatalogError(f"Product id {product_id} not found")
    return self._items.pop(product_id)

Aim for the risky paths covered — not 100% for its own sake.

Section · CI with GitHub Actions

Run it on every push

A suite you forget to run protects nothing. CI runs it automatically on every push and pull request — nobody has to remember.

2.1 · Tests only help if they run

The machine remembers, so nobody has to

A green suite on your laptop helps no one when a teammate pushes a breaking change and never runs it.

you:       suite green on your laptop            — protects only you
teammate:  pushes a breaking change, never runs it   — nobody catches it
CI:        runs the suite on every push           — the machine remembers

CI runs the suite automatically on every push and pull request.

2.2 · The workflow file

A YAML file GitHub runs for you

A file in .github/workflows/ tells GitHub: on a push, set up Python, install, run pytest.

on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v5     # this project is uv-based
      - uses: actions/setup-python@v5
      - run: uv sync                     # installs the dev-group tools
      - run: uv run pytest

2.3 · The matrix + the green check

One suite, several Python versions

A matrix runs the suite across several Python versions at once — catch a 3.10-only break before users do. The result is a ✓ or ✗ on the pull request itself.

strategy:
  fail-fast: false                 # one version failing must not cancel the rest
  matrix:
    python-version: ["3.10", "3.11", "3.12"]

Section · Make it a gate

Red can't merge

Turn "tests exist" into "red can't reach main": artifacts, a threshold, a gate.

3.1 · Reports — artifacts you can open

CI's console scrolls away — produce files

Produce files: an HTML report (browsable) + JUnit XML (other tools read it), uploaded as build artifacts.

- run: pytest --cov --html=report.html --junitxml=results.xml
- uses: actions/upload-artifact@v4
  if: always()                     # upload even when tests FAIL — you need it most then
  with: { name: report, path: report.html }

3.2 · Fail under a threshold

Make coverage a rule, not a vibe

--cov-fail-under fails the build if coverage drops below the line — so a new untested feature can't sneak in.

pytest --cov=catalog --cov-fail-under=80     # exit 1 if below 80%
🔮
Predict: your suite sits at 83%. A teammate adds a feature with no tests and it drops to 78% — what does CI do?

3.3 · The payoff — a quality gate

Red can't reach main

Require the check to pass before merge, and red can't reach main. Your suite stopped being "tests that exist" and became a gate.

push → CI runs pytest + coverage →   ✓ merge allowed   /   ✗ blocked
