From a dict to a class

Fields and behavior, in one typed object

Module 2 left an account as a dict + loose insert/update/fetch functions — and a mixed-value dict that was hard to type. The fix: a class.

acct = {"id": 1, "owner": "Ada", "balance": 1500.0}   # M2: a bare dict
ada  = BankAccount(1, "Ada", 1500.0)                  # M3: an object with methods

Build by hand → shortcut with @dataclass → specialize with inheritance → expose over HTTP. This class is the core every caller talks to.

OOP — the class by hand

why → class / __init__ / self → attributes → methods
Why now: the app's core must protect its own data — a dict can't.

1.1 · Why a class

A dict has data but no guardrails

Nothing stops balance = -50 or a typo'd key. A class bundles the data and the operations that protect it.

acct = {"owner": "Ada", "balance": 1500.0}   # data only — deposit/withdraw live elsewhere

1.2 · class + __init__ + self

Blueprint → object

A class is a blueprint; an object is one thing built from it. __init__ runs once and sets up its data. self is that object.

class BankAccount:
    def __init__(self, id, owner, balance):
        self.id = id; self.owner = owner; self.balance = balance

ada = BankAccount(1, "Ada", 1500.0)   # __init__ runs, returns the object

1.3 · Attributes

The data on an object

Per-object data set in __init__. Read/write with a dot — a typo is an error, not a silent None.

ada.balance          # 1500.0   (vs d["balance"])
ada.owner = "Ada K." # set with the dot too

1.4 · Methods

Behavior, with guardrails

A method is a function inside the class; its first parameter is self. A method can raise to enforce a rule, so the data can't go invalid.

class BankAccount:
    ...
    def withdraw(self, amount):
        if amount > self.balance:
            raise ValueError("insufficient funds")   # the rule lives with the data
        self.balance -= amount

@dataclass — the clean record

boilerplate → @dataclass → the data-structure step → defaults → methods
Why now: the core's shape is right — kill the boilerplate so the record stays honest.

2.1 · That's a lot of boilerplate

Six fields, written by hand

Writing __init__ (plus a readable print, plus ==) by hand for every field is tedious and easy to get wrong.

def __init__(self, id, owner, account_type, balance, is_active, tags):
    self.id = id; self.owner = owner; ...   # ...repeat for every field

2.2 · @dataclass — boilerplate for free

Just list the typed fields

Put @dataclass on top. You get __init__, a readable __repr__, and __eq__ (== by value) — written for you.

@dataclass
class BankAccount:
    id: int; owner: str; balance: float

print(BankAccount(1, "Ada", 1500.0))   # BankAccount(id=1, owner='Ada', balance=1500.0)
@
The @ is a decorator — it hands your class to a framework. In this course you use decorators; you never write your own.

2.3 · The next data structure

variable → list & dict → dataclass

The progression: variable (M1) → list & dict (M2) → dataclass (M3) — a typed, fixed-shape record. Exactly the fix for M2's untypeable mixed-value dict.

@dataclass
class BankAccount:
    id: int
    owner: str
    balance: float        # each field named AND typed — no more dict[str, object]

2.4 · Defaults & the tags field

Mutable defaults need a factory

Give a field a default after the typed ones. For a mutable default (a list), use field(default_factory=list) — never tags: list = [] (the M1 trap).

@dataclass
class BankAccount:
    id: int; owner: str; balance: float
    is_active: bool = True
    tags: list[str] = field(default_factory=list)

2.5 · Behavior still lives on it

Clean data and behavior, together

A dataclass is still a class — your methods sit right alongside the fields.

@dataclass
class BankAccount:
    id: int; owner: str; balance: float = 0.0
    def withdraw(self, amount):
        if amount > self.balance: raise ValueError("insufficient funds")
        self.balance -= amount

Inheritance

is-a → inherit → override + super() → Pydantic
Why now: specialize the core without rewriting it — the is-a that powers Pydantic.

3.1 · Why inherit — is-a

Reuse everything, change the difference

A SavingsAccount is a BankAccount that also earns interest. Inheritance reuses everything the parent has and adds just the difference.

@dataclass
class SavingsAccount(BankAccount):
    rate: float = 0.04        # plus everything BankAccount already has

3.2 · A subclass inherits everything

Fields and methods, for free

SavingsAccount(BankAccount) gets the parent's fields and methods — withdraw, balance, all of it — without rewriting them.

s = SavingsAccount(1, "Ada", 1500.0)
s.withdraw(100)        # BankAccount's method, inherited as-is

3.3 · Override + super()

Specialize, then reuse the parent

Redefine a method to specialize it; call super().<method>() to reuse the parent's logic and add to it.

class SavingsAccount(BankAccount):
    def withdraw(self, amount):
        super().withdraw(amount)   # reuse the parent's guardrail
        self.balance -= 1          # then apply a savings withdrawal fee

3.4 · You'll see this again — Pydantic

The same is-a, plus validation

Day 2's models are classes that inherit from Pydantic's BaseModel — same is-a mechanism, plus runtime validation.

class Product(BaseModel):   # is-a BaseModel → gets validation for free
    id: int
    name: str

FastAPI — the app's front door

what it is → why an API → why FastAPI → build → run → where it sits → AI apps

4.1 · What FastAPI is

Functions become HTTP endpoints

FastAPI is a web framework: it turns your Python functions into URLs other programs can call over the network.

 your Python function  ─FastAPI─▶  GET http://localhost:8000/products

It's the front door to your app — the core stays pure Python; FastAPI lets the outside world in.

4.2 · Why an API?

Callers aren't always Python

A browser, a mobile app, another team's service can't import catalog — they speak HTTP. The API is the shared doorway to your engine.

 browser ┐
 mobile  ├─ HTTP ─▶ server.py ─▶ ProductCatalog ─▶ files
 service ┘         (one door)      (one core)         (data)

One core, many callers. Change the core freely; the doorway stays the same.

4.3 · Why FastAPI

Your type hints start doing work

The hints you've written since M1 become real machinery — for free:

  • Validation — a route typed product_id: int rejects /products/abc with a clean 422, before your code runs
  • Docs — an interactive /docs page, generated from your signatures
  • Pydantic-powered — the Day-2 models become the request / response contract
  • The Python standard for APIs — and for AI / ML services (model endpoints, RAG, agent backends)

4.4 · A route is a function with @app.get

Put the decorator on top

FastAPI turns a function into a web endpoint: put @app.get("/path") above it, and it runs when a request hits that path.

app = FastAPI()

@app.get("/accounts")
def list_accounts():
    return ACCOUNTS
@
That @ is a decorator again — same idea as @dataclass. You'll meet it twice more: @pytest.fixture (Day 3), @agent.tool (Day 4).

4.5 · Build the server — GET & POST

One function per route

Return your objects and FastAPI serialises them to JSON; take a path parameter to fetch one.

@app.get("/accounts/{id}")
def get_account(id: int):
    return fetch(STORE, id)

@app.post("/accounts")
def create(acct: dict):
    insert(STORE, acct); return acct

4.6 · Run it — uvicorn + /docs

Interactive docs, for free

Start the server, then hit it from a browser or curl. This is the end-of-Day-1 artifact: a running API serving a catalog you built.

uv sync                                # once — installs FastAPI
uv run uvicorn catalog.server:app --reload
curl localhost:8000/accounts/1
# then open http://localhost:8000/docs

4.7 · The shape of a real Python app

Three files, three layers

The modules you wrote are a layered application — each one job, each layer talking only to the next.

 HTTP ─▶ server.py   API layer    — routes, status, JSON
           │
           ▼
        models.py   domain core  — Product, Catalog, rules
           │
           ▼
        storage.py  persistence  — save / load JSON, CSV

The layering is the lesson — swap FastAPI for a CLI, or JSON for a database, and the core never changes.

4.8 · The same shape builds AI apps

An AI app is a FastAPI backend + an LLM

Your catalog server is exactly the backend an AI feature calls — and tool-calling is the pattern you just learned:

 user ─▶ FastAPI backend ─▶ LLM ─▶ picks a tool
                 ▲                        │
                 └──  your catalog  ◀─────┘
@
@app.get exposes your function to the web; @agent.tool exposes it to an LLM. The Pydantic types become the tool's schema. Same move.
