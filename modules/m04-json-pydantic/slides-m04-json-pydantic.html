From @dataclass to Pydantic

the problem → BaseModel → validated on construction
Why now: the server's front door is open — something has to check what walks in.

1.1 · The problem

Your annotations don't run — in TS or in Python

You already have the right instinct: a TS interface is erased — it guards nothing at runtime. Python's hints are exactly the same. Decoration.

interface Account { balance: number }      // TS: erased at compile time — checks nothing

BankAccount(id=1, owner="", balance=-50)   # Python: accepted silently — the hint checked nothing

Pydantic is where that stops being true. M3 promised your hints would start doing work — this is the module where they actually do. (And the bad value doesn't fail here; it fails three functions later, somewhere confusing.)

1.2 · Pydantic

Same fields, one base class

Subclass BaseModel and list the same typed fields. That's the only structural change — the is-a inheritance you met in M3, now earning its keep.

from pydantic import BaseModel

class BankAccount(BaseModel):
    id: int
    owner: str
    balance: float

1.3 · Validated on construction

Bad data raises immediately

A BaseModel checks types when you build it: good data becomes an object, bad data raises ValidationError — at the boundary, not deep in your code.

BankAccount(id=1, owner="Ada", balance=1500.0)    # ok
BankAccount(id="x", owner="Ada", balance=1500.0)  # ValidationError: id is not an int

Validation rules

Field constraints → reading a ValidationError
Why now: the type is only half the rule — "a float" isn't "a price".

2.1 · Field

Rules beyond the type

A type says "a float"; Field says "a float ≥ 0". Constraints live on the field and run on construction — the rule sits with the data, not in some validator you have to remember to call.

from pydantic import Field

class BankAccount(BaseModel):
    owner: str = Field(min_length=1)
    balance: float = Field(ge=0)        # ge = "greater than or equal to"
≡
Used zod? Field(min_length=1) is z.string().min(1). Same idea, same place — the rule rides on the field.

2.2 · Reading a ValidationError

Structured, not a wall of text

For every bad field it tells you where (loc), what (msg), and why (type). That list is machine-readable — M6 drops it straight into an import report.

try:
    BankAccount(id=1, owner="", balance=-5)
except ValidationError as e:
    e.errors()   # [{'loc': ('owner',), 'msg': '...', 'type': 'string_too_short'}, ...]

Section

model_validate / model_dump

dict → model → dict, with coercion
Why now: data arrives as dicts and leaves as dicts — these two are the doors.

3.1 · model_validate

A dict becomes a model

An HTTP body or a file row arrives as a plain dict. model_validate turns it into a validated object — or raises.

BankAccount.model_validate({"id": 2, "owner": "Lin", "balance": 800.0})   # dict → model
→
Got a JSON string rather than a dict? model_validate_json(raw) — same gate, one step earlier, no json.loads first. On Day 4 the LLM answers with text: that's the line that turns text into a checked object.

3.2 · model_dump

A model becomes a dict

The other direction — back to a plain dict, ready for JSON and the wire.

acct.model_dump()   # {'id': 2, 'owner': 'Lin', 'balance': 800.0}

These two are on every line of Modules 5–6. Learn them here and the rest of Day 2 is assembly.

3.3 · Coercion

Where your zod instinct is wrong

zod is strict — z.number().parse("800") throws. Pydantic is lax — it converts: "800.0"→float, "true"→bool, "2"→int. Same intent, opposite default. This is the one place your JS instinct will mislead you.

# zod:  z.number().parse("800")  ->  ZodError
BankAccount.model_validate({"id": "2", "owner": "Lin", "balance": "800.0"})  # -> 800.0 ✓

This is why a CSV of strings feeds the API with no manual parsing (M6's through-line) — and why ConfigDict(strict=True) exists for when you want zod's behaviour back.

Pydantic in FastAPI

typed body → 422 → response_model → docs
Why now: the model you just built is the API contract — plug it in and the server enforces it.

4.1 · A typed body

The route validates itself

Type the route parameter as your model. FastAPI validates the incoming JSON and returns an automatic 422 with field errors — before your code runs. The manual check you wrote yesterday just disappears.

@app.post("/accounts")
def create(account: BankAccount):   # FastAPI validates the body into a BankAccount
    ...

4.2 · response_model

Typed output, too

Declare what a route returns; FastAPI serialises the model to JSON and documents the shape. The contract now runs both ways.

@app.get("/accounts/{id}", response_model=BankAccount)
def get_account(id: int):
    ...

4.3 · /docs, from the schema

The model is the schema

FastAPI's /docs now shows every field, type, and constraint — and lets you try the API in the browser. No extra work. One model, three jobs: validation, serialisation, documentation.

On Day 4 that same schema is what an LLM reads to learn how to call your tool. You write it once, here.

Section

The model is data

model_json_schema → description= → optional fields
Why now: §4 showed the schema rendered as docs. Now take it as data — that's the handle Day 4 gives an LLM.

5.1 · model_json_schema()

The schema, as a plain dict

§4.3 claimed the model is the schema. Here it is — not a docs page, a dict. Look closely: the Field rules from §2.1 became JSON Schema keywords, for free.

Product.model_json_schema()
# {'type': 'object', 'title': 'Product',
#  'properties': {'id':    {'type': 'integer', 'minimum': 1},      # <- Field(ge=1)
#                 'name':  {'type': 'string', 'minLength': 1},     # <- Field(min_length=1)
#                 'price': {'type': 'number', 'minimum': 0}, ...}, # <- Field(ge=0)
#  'required': ['id', 'name', 'category', 'price']}

/docs is rendered from this. And on Day 4 you hand this exact dict to an LLM so it knows how to call your function. One model, three consumers.

5.2 · description=

Documentation that is prompt text

The same Field() from §2.1, doing a different job. The description rides into the schema (§5.1) — and on Day 4 the LLM reads it to decide what to pass.

class CatalogQuery(BaseModel):
    max_price: float | None = Field(
        default=None, ge=0,
        description="Upper price bound in INR, or null for no bound.",
    )
# -> schema: {'description': 'Upper price bound in INR, or null for no bound.',
#             'anyOf': [{'type': 'number', 'minimum': 0}, {'type': 'null'}]}

The punchline of Day 2 → Day 4: a docstring the machine obeys. Write it badly and the LLM calls your tool badly.

5.3 · Optional fields

`| None` — may be absent, still checked

in_stock: bool = True was a default. This is different: the field may legitimately be nothing at all. str | None and Optional[str] are the same thing — | None is the modern spelling.

CatalogQuery.model_validate({})                    # all None — fine
CatalogQuery.model_validate({"max_price": 2000})   # -> 2000.0
CatalogQuery.model_validate({"max_price": -5})     # ValidationError — ge=0 still applies

Optional means "may be absent", not "unchecked". Day 4's CatalogQuery is entirely optional fields — ask for "cheap electronics" and you named no product, so name_contains is None.
