HTTP with requests

verbs → status codes → where's the await? → Session + timeout → auth
Why now: the rules first — §2 is where you enforce them, and you can't enforce what you can't name.

1.1 · Verbs

Which are safe to re-send?

A retry re-sends the identical request — so the only question is: does sending it twice do harm? Idempotent = same end state no matter how many sends.

 DELETE /accounts/5   ×2  → still just "deleted"   (idempotent → safe)
 POST   /accounts {…} ×2  → TWO accounts created   (not idempotent)

GET/PUT/DELETE are idempotent; POST isn't. Hold that thought — §2.2 has to answer for it.

1.2 · Status codes

Who broke — you, or the server?

A 4xx means the server understood you and refused: your request is wrong, so sending it again earns the same refusal, forever. A 5xx or a dropped connection is the server stumbling — the same request might work a moment later.

 POST {bad json} → 422 → retry → 422 → 422 …   never recovers
 GET  /accounts  → 503 → retry → 200           retry won

So: retry 5xx + network errors, never 4xx. That one line is the whole policy — §2.2 makes the code obey it.

1.3 · Wait — where's the await?

There isn't one. That's not a typo.

You're about to see resp = session.get(url) with no await. requests is blocking: the line stops, waits for the server, then hands you the answer.

# JS — the network is always async
const r = await fetch(url);  const data = await r.json();

# Python + requests — the line just waits
r = session.get(url);        data = r.json()

Why that's right here: there's no UI thread to starve — this is a script, not a browser. Blocking is the simple choice, and simple is correct for scripts, tests, and imports.

1.3b · Python has async too

It's a choice, not an absence

Async HTTP exists — httpx, aiohttp — and you'd reach for it when one process must hold thousands of connections at once. That's a server's problem, not ours.

async def main():                       # the async flavour, for reference
    async with httpx.AsyncClient() as c:
        r = await c.get(url)            # there's your await
↩
Day-1 callback: your FastAPI routes were def, not async def — and they worked. FastAPI runs a plain def route in a threadpool so blocking code can't stall the server. You picked the simple spelling without knowing it.

Rule of thumb: scripts & tests → sync requests. Servers under load → async.

1.4 · Session + timeout

One connection, and never wait forever

Each bare requests.get(...) opens a new TCP connection and forgets your headers. A Session keeps one alive and carries them — it is axios.create({ baseURL, headers }): configure once, reuse everywhere. And timeout is not optional — omit it and a server that never replies blocks your thread forever.

s = requests.Session()       # one connection, reused; headers remembered
s.get(url)                   # ⚠️ no timeout → hangs forever if the server stalls
s.get(url, timeout=5)        # ✓ raises after 5s instead of hanging

1.5 · Auth

A token in a header, from the environment

Auth is just a header you set once on the session; every later call carries it. The trap is where the secret lives — a token in the URL lands in server logs and browser history; a token in source lands in git. Both leak forever.

# ❌ token in the URL → logged everywhere it travels
s.get(f"{url}?api_key={TOKEN}")
# ✓ token in a header, value from an env var → never logged, never committed
s.headers["Authorization"] = f"Bearer {os.environ['TOKEN']}"

The APIClient

_request funnel → the retry loop → typed/validated returns
Why now: §1 gave the rules — one funnel is where you enforce all of them at once.

2.1 · One _request funnel

Write it once, or get it wrong five times

Five CRUD methods each repeating url-join + timeout + error-check = five places to slip (and one route quietly missing a timeout). Funnel it: every public method becomes a one-liner that can't forget.

def _request(self, method, path, **kw):
    resp = self._session.request(method, self.base_url + path, timeout=5, **kw)
    if not resp.ok:
        raise APIError(resp.status_code, resp.text)   # non-2xx → a clean error
    return resp

def delete_account(self, id):  self._request("DELETE", f"/accounts/{id}")   # one line

2.2 · The retry loop

§1.2's rule, made structural

Because every call funnels through _request, the retry goes in one place. Look at what the except catches: only network faults. A 4xx never reaches it — it becomes an APIError, raised after the loop. The rule isn't a convention here; it's the shape of the code.

for attempt in range(1, 4):
    try:
        resp = self._session.request(method, url, **kw)
        break                                   # got an answer — done looping
    except (ConnectionError, Timeout):          # ONLY transient faults
        if attempt == 3: raise                  # out of tries — give up
        time.sleep(0.2)                         # else wait, go again

if not resp.ok:
    raise APIError(resp.status_code, ...)       # a 422 lands HERE — never retried

2.2b · The honest caveat

But §1.1 said POST isn't idempotent…

Right — and this loop retries every verb, create_product included. A timeout can mean the server did the work and the reply got lost; retrying then sends it twice. We accept that here for one specific reason:

 our ids are client-supplied → a double POST hits the
 duplicate-id rule → 409, recorded in the report (M6)
 …not a silent second account

The catalog's own rule is what makes retry-everything safe. Without that, you'd scope the loop to idempotent verbs — real clients do exactly that, or send an idempotency key. Know which world you're in.

2.3 · Typed returns

Validate at the boundary

Return the raw dict and a server that drops balance blows up three functions later, mysteriously. Parse into the M4 model right here — a bad response fails now, with a field-level error, and callers get real objects.

def list_accounts(self) -> list[BankAccount]:
    data = self._request("GET", "/accounts").json()       # list of dicts off the wire
    return [BankAccount.model_validate(r) for r in data]  # validated → fails here if malformed

This is what "validating API responses" means in practice.
